# Code Reviewer

## 概述

对当前 git 变更进行结构化审查，重点关注 SOLID 原则、架构设计、清理候选项和安全风险。默认仅输出审查结果，除非用户明确要求实施修改。

## 严重程度

| 等级   | 级别     | 描述                                              | 处理方式                         |
| ------ | -------- | ------------------------------------------------- | -------------------------------- |
| **P0** | Critical | 安全漏洞、数据丢失风险、正确性缺陷               | 必须阻止合并                     |
| **P1** | High     | 逻辑错误、严重 SOLID 违规、性能退化              | 合并前应修复                     |
| **P2** | Medium   | 代码异味、可维护性问题、轻微 SOLID 违规          | 当前 PR 修复或创建 follow-up     |
| **P3** | Low      | 风格、命名、轻微建议                              | 可选优化                         |

## 工作流程

### 1) 预检上下文

- 使用 `git status -sb`、`git diff --stat` 和 `git diff` 确定变更范围。
- 必要时使用 `rg` 或 `grep` 查找相关模块、调用方和接口契约。
- 识别入口点、职责边界和关键路径（认证、支付、数据写入、网络请求）。
- 若变更涉及公共函数、共享类型、全局状态等，使用 `rg` 追踪所有调用方，确认上下游是否同步更新。

**边界情况：**
- **无变更**：若 `git diff` 为空，通知用户并询问是否审查 staged changes 或指定 commit range。
- **大型 diff（>500 行）**：先按文件汇总，再按模块/功能区域分批审查。
- **混合关注点**：按逻辑功能分组发现项，而非按文件顺序排列。

### 2) SOLID + 架构异味

- 加载 `references/solid-checklist.md` 获取检查提示。
- 检查以下违规：
  - **SRP**：模块承载了不相关的职责。
  - **OCP**：新增行为需要频繁修改已有代码，而非通过扩展点实现。
  - **LSP**：子类打破了基类预期，或需要类型检查才能正常工作。
  - **ISP**：接口方法过多，大部分实现者用不到。
  - **DIP**：高层逻辑直接依赖低层实现。
- 提出重构建议时，需解释*为什么*能改善内聚/耦合，并给出最小化、安全的拆分方案。
- 若重构较复杂，提出增量迁移计划而非大规模重写。

### 3) 清理候选项 + 迭代计划

- 加载 `references/removal-plan.md` 获取模板。
- 识别未使用、冗余或已关闭 feature flag 的代码。
- 区分**可立即安全删除**与**需延期处理并制定计划**的情况。
- 提供包含具体步骤和验证检查点（测试/指标）的后续计划。

### 4) 安全与可靠性扫描

- 加载 `references/security-checklist.md` 获取检查覆盖范围。
- 检查以下风险：
  - XSS、注入攻击（SQL/NoSQL/命令注入）、SSRF、路径遍历
  - AuthZ/AuthN 缺陷、缺失的租户隔离检查
  - 密钥泄露或 API Key 暴露在日志/环境变量/文件中
  - 速率限制缺失、无限循环、CPU/内存热点
  - 不安全的反序列化、弱加密算法、不安全的默认配置
  - **竞态条件**：并发访问、check-then-act 模式、TOCTOU、缺失锁机制
- 同时指出**可利用性**和**影响范围**。

### 5) 代码质量扫描

- 加载 `references/code-quality-checklist.md` 获取检查覆盖范围。
- 检查以下问题：
  - **错误处理**：异常吞没、过于宽泛的 catch、缺失错误处理、异步错误未捕获
  - **性能**：N+1 查询、热点路径中的 CPU 密集操作、缓存缺失、无限增长的内存
  - **边界条件**：null/undefined 处理、空集合、数值边界、Off-by-one 错误
- 标记可能导致静默失败或生产事故的问题。

### 6) 输出格式

按以下结构组织审查结果：

```markdown
## Code Review Summary

**审查文件**：X 个文件，Y 行变更
**整体评估**：[APPROVE / REQUEST_CHANGES / COMMENT]

---

## 发现项

### P0 - Critical
（无 或 列表）

### P1 - High
1. **[file:line]** 简要标题
  - 问题描述
  - 修复建议

### P2 - Medium
2. （跨分区连续编号）
  - ...

### P3 - Low
...

---

## 清理/迭代计划
（如适用）

## 补充建议
（可选优化，不阻塞合并）
```

**行内注释**：针对特定文件的发现项使用以下格式：
```
::code-comment{file="path/to/file.ts" line="42" severity="P1"}
问题描述及修复建议。
::
```

**无问题审查**：若未发现问题，需明确说明：
- 审查了哪些内容
- 未覆盖的区域（例如"未验证数据库迁移脚本"）
- 残余风险或建议的后续测试

### 7) 后续步骤确认

展示审查结果后，询问用户如何处理：

```markdown
---

## 后续步骤

共发现 X 个问题（P0: _，P1: _，P2: _，P3: _）。

**你希望如何处理？**

1. **全部修复** - 实施所有建议的修复
2. **仅修复 P0/P1** - 处理 Critical 和 High 级别问题
3. **修复指定项** - 告诉我需要修复哪些问题
4. **不做修改** - 审查完成，无需实施

请选择一个选项或提供具体指示。
```

**重要**：在用户明确确认之前，不要实施任何修改。这是审查优先的工作流。
