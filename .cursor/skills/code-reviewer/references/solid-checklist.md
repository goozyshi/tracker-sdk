# SOLID 异味检查提示

## SRP（单一职责原则）

- 文件承载了不相关的关注点（例如 HTTP + DB + 领域规则混在一个文件中）
- 大型类/模块内聚性低，存在多个变更原因
- 函数编排了多个不相关的步骤
- God Object：对系统了解过多的上帝对象
- **自问**："这个模块会因为哪一个唯一的原因而变更？"

## OCP（开闭原则）

- 新增行为需要修改大量 switch/if 分支
- 功能扩展需要修改核心逻辑而非通过扩展实现
- 缺少 plugin/strategy/hook 等变化扩展点
- **自问**："我能否在不修改已有代码的情况下新增一个变体？"

## LSP（里氏替换原则）

- 子类检查具体类型或对基类方法抛出异常
- 重写方法弱化了前置条件或强化了后置条件
- 子类忽略或空实现父类行为
- **自问**："我能否替换任意子类而不影响调用方？"

## ISP（接口隔离原则）

- 接口方法过多，大部分实现者用不到
- 调用方为了少量需求而依赖臃肿的接口
- 接口方法存在空实现或 stub 实现
- **自问**："所有实现者是否都使用了所有方法？"

## DIP（依赖倒置原则）

- 高层逻辑直接依赖具体的 IO、存储或网络类型
- 硬编码实现而非使用抽象或依赖注入
- import 链将业务逻辑与基础设施耦合
- **自问**："我能否在不修改业务逻辑的情况下替换实现？"

---

## 常见代码异味（SOLID 之外）

| 异味                       | 表现                               |
| -------------------------- | ---------------------------------- |
| **Long Method**            | 函数超过 30 行，多层嵌套           |
| **Feature Envy**           | 方法使用其他类的数据多于自身的数据 |
| **Data Clumps**            | 同一组参数反复一起传递             |
| **Primitive Obsession**    | 使用 string/number 代替领域类型    |
| **Shotgun Surgery**        | 一处变更需要修改多个文件           |
| **Divergent Change**       | 一个文件因多个不相关的原因频繁变更 |
| **Dead Code**              | 不可达或从未被调用的代码           |
| **Speculative Generality** | 为假设的未来需求而创建的抽象       |
| **Magic Numbers/Strings**  | 硬编码值未使用命名常量             |

## 重构启发式规则

1. **按职责拆分，而非按体量** - 小文件同样可能违反 SRP
2. **仅在需要时引入抽象** - 等待第二个使用场景出现
3. **保持重构增量进行** - 先隔离行为再迁移
4. **优先保持行为不变** - 重构前先补充测试
5. **按意图命名** - 如果命名困难，说明抽象可能有问题
6. **组合优于继承** - 继承会产生紧耦合
7. **让非法状态不可表示** - 用类型约束不变量
